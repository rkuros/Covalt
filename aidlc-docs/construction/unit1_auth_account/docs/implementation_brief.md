# Unit 1: 認証・アカウント管理 -- 実装ブリーフ

## 1. コンポーネント一覧

### エンティティ

| エンティティ | 属性 | 責務 |
|---|---|---|
| **OwnerAccount** | `ownerId`, `email`, `passwordHash`, `role`("owner"), `status`(有効/無効), `createdAt`, `updatedAt` | オーナーの認証情報とアカウント状態のライフサイクルを管理する。ログイン認証・アカウント有効化/無効化の判定主体。 |
| **AdminAccount** | `adminId`, `email`, `passwordHash`, `role`("admin"), `createdAt` | システム管理者の認証情報を保持する。オーナーアカウントの作成・管理操作の実行主体。 |
| **PasswordResetToken** | `tokenId`, `ownerId`, `token`, `expiresAt`, `usedAt` | パスワードリセット要求を表す。有効期限と使用済みフラグによりワンタイム性を保証する。 |
| **Session** | `sessionId`, `ownerId`, `token`, `createdAt`, `expiresAt` | ログイン済みセッションを表す。タイムアウト判定と認証トークン検証の根拠となる。 |

### 値オブジェクト

| 値オブジェクト | 属性 | 責務 |
|---|---|---|
| **EmailAddress** | `value` | メールアドレスの形式バリデーションを保証する。 |
| **HashedPassword** | `value` | パスワードのハッシュ化済み表現を保持する。平文パスワードとの照合責務を持つ。 |
| **AuthToken** | `value` | Bearer トークン文字列を表す。トークンの生成・解析に使われる。 |
| **AccountStatus** | `value`(ACTIVE / DISABLED) | アカウントの有効/無効状態を型安全に表現する。 |
| **Role** | `value`(owner / admin) | ユーザーのロール種別を表す。認証検証レスポンスで返却される。 |

### ドメインサービス

| サービス | 責務 |
|---|---|
| **AuthenticationService** | メールアドレス + パスワードによるログイン認証、セッション発行、ログアウト処理を行う。 |
| **TokenVerificationService** | Bearer トークンを受け取り、セッションの有効性とアカウント状態を検証して認証情報(ownerId, email, role)を返す。他ユニットからの認証検証リクエスト(POST /api/auth/verify)の中核ロジック。 |
| **PasswordResetService** | パスワードリセットトークンの生成・検証、新パスワードの設定を行う。リセットメール送信を起動する。 |
| **AccountProvisioningService** | システム管理者による新規オーナーアカウントの作成、初期設定メールの送信起動を行う。 |
| **AccountManagementService** | オーナーアカウント一覧の取得、有効/無効の切り替えを行う。 |

## 2. Provider API 仕様サマリ

PACT 契約 (`unit4-unit1-auth.pact.json`) に基づき、Unit 1 が Provider として公開する API は以下の通り。Unit 3, 4, 6, 7 が Consumer として同一契約に準拠する。

### `POST /api/auth/verify` -- 認証トークン検証

| シナリオ | Provider State | リクエスト | レスポンス Status | レスポンス Body |
|---|---|---|---|---|
| 有効なトークンで認証成功 | 有効なオーナーアカウントが存在する | `Authorization: Bearer {valid-token}` | **200** | `{ ownerId: string, email: string, role: "owner" \| "admin" }` |
| 無効なトークンで認証失敗 | トークンが無効である | `Authorization: Bearer {invalid-token}` | **401** | `{ error: "UNAUTHORIZED", message: "認証トークンが無効です" }` |
| 無効化アカウントで認証拒否 | アカウントが無効化されている | `Authorization: Bearer {valid-token-disabled}` | **403** | `{ error: "ACCOUNT_DISABLED", message: "アカウントが無効化されています" }` |

**マッチングルール (200 レスポンス)**:
- `ownerId` -- 型一致 (string)
- `email` -- 型一致 (string)
- `role` -- 正規表現 `^(owner|admin)$`

## 3. ビジネスルール

各ユーザーストーリーのアクセプタンスクライテリアから抽出したビジネスルールを以下に整理する。

### 認証 (US-S01)

| # | ルール |
|---|---|
| BR-01 | メールアドレスとパスワードの組み合わせが正しい場合のみログインを許可する |
| BR-02 | 誤った認証情報によるログイン試行は拒否し、エラーメッセージを返す |
| BR-03 | ログイン成功後、セッションを発行しダッシュボード(予約一覧)へ遷移させる |
| BR-04 | ログアウト操作によりセッションを無効化する |
| BR-05 | 一定時間操作がない場合、セッションを自動タイムアウトさせる |

### パスワードリセット (US-S02)

| # | ルール |
|---|---|
| BR-06 | 登録済みメールアドレスに対してパスワードリセットリンクを送信する |
| BR-07 | リセットリンクには有効期限を設定する（期限切れのリンクは使用不可） |
| BR-08 | リセットリンクから新しいパスワードを設定できる |

### アカウント初期登録 (US-S03)

| # | ルール |
|---|---|
| BR-09 | システム管理者のみがオーナーアカウントを新規作成できる |
| BR-10 | アカウント作成時、オーナーに初期設定用メールを送信する |
| BR-11 | オーナーは初期設定メール内リンクからパスワードを設定して利用を開始する |

### アカウント管理 (US-S04)

| # | ルール |
|---|---|
| BR-12 | システム管理者はオーナーアカウントの一覧を取得できる |
| BR-13 | システム管理者はアカウントを有効/無効に切り替えられる |
| BR-14 | 無効化されたアカウントではログインおよび認証トークン検証が拒否される（PACT 契約: 403 ACCOUNT_DISABLED） |

## 4. 横断的関心事

`cross_cutting_security.md` より、Unit 1 に関係する項目を以下に整理する。

### US-S06: 個人情報の適切な取り扱い (Must)

| # | 要件 | Unit 1 での適用 |
|---|---|---|
| CC-01 | API 通信はすべて HTTPS 経由で行う | 認証 API (`/api/auth/verify`) を含むすべてのエンドポイントに適用 |
| CC-02 | オーナーは自分の顧客情報のみアクセスでき、他オーナーの情報にはアクセスできない | 認証トークン検証で返却する `ownerId` が、下流ユニットでのテナント分離の根拠となる。Unit 1 はこの `ownerId` を正確に返す責務を持つ |
| CC-03 | 顧客情報エクスポート時のアクセス制御 | Unit 1 が発行する認証情報を基に下流ユニットが制御する |

### US-S07: システムの可用性 (Must)

| # | 要件 | Unit 1 での適用 |
|---|---|---|
| CC-04 | システムエラー発生時、顧客にエラーメッセージを表示しオーナーに通知する | 認証基盤の障害時、適切なエラーレスポンスを返す |

## 5. 依存関係

### Unit 1 が依存する外部ユニット

| 依存先 | 内容 |
|---|---|
| (外部サービス) メール送信基盤 | パスワードリセットリンク送信 (US-S02)、初期設定メール送信 (US-S03) |

### Unit 1 に依存するユニット (Consumer)

| Consumer | エンドポイント | 用途 | 連携 ID |
|---|---|---|---|
| Unit 3 (スケジュール管理) | `POST /api/auth/verify` | 認証トークン検証 | A1 |
| Unit 4 (予約管理) | `POST /api/auth/verify` | 認証トークン検証 | A1 |
| Unit 6 (顧客情報管理) | `POST /api/auth/verify` | 認証トークン検証 | A1 |
| Unit 7 (Google カレンダー連携) | `POST /api/auth/verify` | 認証トークン検証 | A1 |

> Unit 1 は他ユニットへの同期 API 呼び出しやイベント発行を行わない。純粋な Provider として機能し、システム全体の認証基盤を提供する。
