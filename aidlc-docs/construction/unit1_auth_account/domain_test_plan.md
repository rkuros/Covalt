## 1. 値オブジェクト (Value Objects)

### 1.1 EmailAddress

- [ ] **正常系**: 有効なメールアドレス形式（例: `owner@example.com`）で生成できること
- [ ] **正常系**: サブドメイン付きメールアドレス（例: `user@mail.example.com`）で生成できること
- [ ] **正常系**: `+` 記号付きメールアドレス（例: `user+tag@example.com`）で生成できること
- [ ] **正常系**: 同一の値を持つ EmailAddress 同士が等価であること（値オブジェクトの等価性）
- [ ] **異常系**: `@` がないメールアドレスで生成に失敗すること
- [ ] **異常系**: ドメイン部分がないメールアドレス（例: `user@`）で生成に失敗すること
- [ ] **異常系**: ローカル部分がないメールアドレス（例: `@example.com`）で生成に失敗すること
- [ ] **異常系**: 空文字で生成に失敗すること
- [ ] **異常系**: null / undefined で生成に失敗すること
- [ ] **境界値**: メールアドレスの最大長に対するバリデーション

## 2. エンティティ (Entities)

### 2.1 OwnerAccount

- [ ] **正常系**: 必要な属性（`ownerId`, `email`, `passwordHash`, `role`, `status`, `createdAt`, `updatedAt`）で生成できること
- [ ] **正常系**: `role` が `"owner"` であること
- [ ] **正常系**: 生成時のデフォルト `status` が `ACTIVE` であること（仕様確認要）
- [ ] **正常系**: ログイン認証 -- 正しいメールアドレスとパスワードの組み合わせで認証成功すること（BR-01）
- [ ] **正常系**: アカウント有効化 -- `status` を `ACTIVE` に切り替えられること（BR-13）
- [ ] **正常系**: アカウント無効化 -- `status` を `DISABLED` に切り替えられること（BR-13）
- [ ] **正常系**: `updatedAt` が状態変更時に更新されること
- [ ] **正常系**: 同一の `ownerId` を持つエンティティが同一と判定されること（エンティティの同一性）
- [ ] **異常系**: ログイン認証 -- 誤ったパスワードで認証失敗すること（BR-02）
- [ ] **異常系**: 無効化されたアカウント（`status` が `DISABLED`）ではログイン認証が拒否されること（BR-14）
- [ ] **異常系**: 必須属性が欠落している場合に生成に失敗すること
- [ ] **境界値**: `ownerId` が空文字の場合の挙動

## 3. ドメインサービス (Domain Services)

### 3.1 AuthenticationService

**責務**: メールアドレス + パスワードによるログイン認証、セッション発行、ログアウト処理

#### ログイン認証

- [ ] **正常系**: 正しいメールアドレスとパスワードでログイン成功し、セッションが発行されること（BR-01, BR-03）
- [ ] **正常系**: 発行されたセッションにトークンが含まれること
- [ ] **正常系**: 発行されたセッションに適切な有効期限が設定されていること
- [ ] **異常系**: 存在しないメールアドレスでログイン失敗すること（BR-02）
- [ ] **異常系**: 誤ったパスワードでログイン失敗すること（BR-02）
- [ ] **異常系**: 無効化されたアカウント（`DISABLED`）でログイン失敗すること（BR-14）
- [ ] **異常系**: メールアドレスが空の場合にエラーとなること
- [ ] **異常系**: パスワードが空の場合にエラーとなること

#### ログアウト

- [ ] **正常系**: ログアウト操作によりセッションが無効化されること（BR-04）
- [ ] **異常系**: 存在しないセッションIDでログアウトを試みた場合の挙動

**[Answer]**

### 3.2 TokenVerificationService

**責務**: Bearer トークンを受け取り、セッションの有効性とアカウント状態を検証して認証情報を返す（PACT 契約 `POST /api/auth/verify` の中核ロジック）

- [ ] **正常系**: 有効なトークンで認証検証が成功し、`ownerId`, `email`, `role` を返すこと（PACT: 200レスポンス）
- [ ] **正常系**: レスポンスの `ownerId` が string 型であること（PACT マッチングルール）
- [ ] **正常系**: レスポンスの `email` が string 型であること（PACT マッチングルール）
- [ ] **正常系**: レスポンスの `role` が `^(owner|admin)$` にマッチすること（PACT マッチングルール）
- [ ] **正常系**: admin アカウントのトークンで検証した場合、`role` が `"admin"` で返ること
- [ ] **正常系**: owner アカウントのトークンで検証した場合、`role` が `"owner"` で返ること
- [ ] **異常系**: 無効なトークンで認証検証が失敗し、401 + `UNAUTHORIZED` エラーが返ること（PACT: 401レスポンス）
- [ ] **異常系**: 無効化されたアカウントのトークンで認証検証が拒否され、403 + `ACCOUNT_DISABLED` エラーが返ること（PACT: 403レスポンス, BR-14）
- [ ] **異常系**: 有効期限切れセッションのトークンで認証検証が失敗すること（BR-05）
- [ ] **異常系**: トークンが空文字の場合にエラーとなること
- [ ] **異常系**: Authorization ヘッダーが存在しない場合の挙動

## 4. PACT 契約準拠テスト (Provider Verification)

**対象契約**: `unit4-unit1-auth.pact.json`（Unit 3, 4, 6, 7 共通）

### 4.1 Provider State: 「有効なオーナーアカウントが存在する」

- [ ] **正常系**: `Authorization: Bearer valid-token-12345` で `POST /api/auth/verify` にリクエストすると、ステータス 200 が返ること
- [ ] **正常系**: レスポンスボディに `ownerId`（string）, `email`（string）, `role`（`^(owner|admin)$`）が含まれること
- [ ] **正常系**: レスポンスヘッダーに `Content-Type: application/json` が含まれること

### 4.2 Provider State: 「トークンが無効である」

- [ ] **正常系**: `Authorization: Bearer invalid-token` で `POST /api/auth/verify` にリクエストすると、ステータス 401 が返ること
- [ ] **正常系**: レスポンスボディに `error: "UNAUTHORIZED"`, `message: "認証トークンが無効です"` が含まれること

### 4.3 Provider State: 「アカウントが無効化されている」

- [ ] **正常系**: `Authorization: Bearer valid-token-disabled` で `POST /api/auth/verify` にリクエストすると、ステータス 403 が返ること
- [ ] **正常系**: レスポンスボディに `error: "ACCOUNT_DISABLED"`, `message: "アカウントが無効化されています"` が含まれること
