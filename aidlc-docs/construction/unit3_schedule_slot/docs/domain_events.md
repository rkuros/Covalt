# Unit 3: ドメインイベント設計

## 設計判断の根拠

- Unit 3 からのドメインイベント発行は現時点で不要（Q12）
- 営業時間変更・休業日設定・スロットステータス変化いずれも他ユニットへの通知不要
- スロットの予約確保・解放は Unit 4 からの同期 API 呼び出しで完結する

---

## 現時点のドメインイベント

**なし**

Unit 3（スケジュール・空き枠管理）は、現時点ではドメインイベントを発行しない。

### 理由

1. **営業時間の変更**: 他ユニットが営業時間変更に反応する要件がない。スロットの再生成はオーナーの操作またはスロット生成サービスの呼び出しにより行われる。

2. **休業日の設定・解除**: 既存予約は維持し新規のみブロックする方針（Q11）のため、Unit 4（予約管理）への通知は不要。既存予約の対応はオーナーの手動オペレーション。

3. **スロットのステータス変化（reserve / release）**: Unit 4 からの同期 API 呼び出しで完結し、呼び出し元に直接レスポンスが返る。非同期イベントによる通知は不要。

---

## 将来の拡張候補

将来的に以下のドメインイベントが必要になる可能性がある。設計上、イベント発行の追加が容易な構造にしておく。

### BusinessHoursChanged（営業時間変更）

| フィールド | 型 | 説明 |
|-----------|------|------|
| ownerId | OwnerId | 対象オーナー |
| dayOfWeek | DayOfWeek | 変更された曜日 |
| timestamp | datetime | イベント発生日時 |

**想定ユースケース**: 営業時間変更時に影響する未来日のスロットを自動再生成する仕組みが必要になった場合。

### ClosedDaySet（休業日設定）

| フィールド | 型 | 説明 |
|-----------|------|------|
| ownerId | OwnerId | 対象オーナー |
| date | SlotDate | 休業日に設定された日付 |
| timestamp | datetime | イベント発生日時 |

**想定ユースケース**: 休業日設定時に Unit 4 が既存予約の顧客に自動通知する仕組みが必要になった場合。

---

## Unit 3 が Consumer として受信するイベント

Unit 3 は現時点で他ユニットからのドメインイベントを受信する要件もない。スロットの reserve / release は Unit 4 からの同期 API 呼び出しで行われる。
