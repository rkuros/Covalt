## 1. マルチテナント分離（オーナーデータの分離）

### 該当要件

> オーナーは自分の顧客情報のみアクセスでき、他のオーナーの情報にはアクセスできない（US-S06）

### Unit 3 への適用

#### 原則

Unit 3 の全データはオーナー単位で管理される。あるオーナーのデータ（営業時間、休業日、スロット）に対して、他のオーナーがアクセスすることはできない。

#### ドメインモデルへの反映

- **BusinessHour**: ownerId を必須属性として保持。照会・更新時に ownerId による絞り込みを行う
- **ClosedDay**: ownerId を必須属性として保持。照会・更新時に ownerId による絞り込みを行う
- **DailySlotList**: ownerId を必須属性として保持。照会・更新時に ownerId による絞り込みを行う

#### 適用レベル

| レイヤー | 対応方針 |
|---------|---------|
| ドメインモデル | 全集約ルートが ownerId を保持。リポジトリインターフェースの全メソッドが ownerId をパラメータに含む |
| アプリケーション層 | 認証トークンから取得した ownerId と操作対象データの ownerId の一致を検証する |
| インフラ層 | リポジトリ実装で ownerId による WHERE 条件を常に適用する |

#### Unit 4 からのアクセス

- Unit 4 が PACT 経由で Unit 3 の API を呼び出す際も、ownerId をクエリパラメータまたはコンテキストとして渡す
- Unit 3 は受け取った ownerId に基づいてデータをフィルタする

## 2. 排他制御（スロットの二重予約防止）

### 該当要件

> 予約の二重登録が発生しないよう、排他制御が実装されている（US-S07）

### Unit 3 への適用

#### 方式

楽観的ロック（version フィールド）を採用（Q8）。

#### ドメインモデルへの反映

- **DailySlotList** 集約ルートに version（Version 値オブジェクト）を保持する
- 集約の更新時に version を検証し、不一致の場合は楽観的ロック例外を発生させる
- version は更新成功のたびにインクリメントする

#### 二重予約防止の仕組み

1. **ドメインルールによる防止**: Slot の status が booked の場合、reserve 操作は SLOT_ALREADY_BOOKED エラーを返す（PACT: 409）
2. **楽観的ロックによる防止**: 同一 DailySlotList への同時更新（2つの予約確保リクエストが同時に同じスロットを狙う場合）を version 検証で検出する
3. **エラーの区別**:
   - Slot が既に booked -> SLOT_ALREADY_BOOKED（ドメインエラー、409）
   - version 不一致 -> 楽観的ロック例外（インフラエラー、409 または 503 として返却）

#### リトライ方針

- 楽観的ロック例外が発生した場合、アプリケーション層でリトライ可能
- リトライ時にスロットの最新状態を再取得し、まだ available であれば再度 reserve を試行

## 3. HTTPS 通信

### 該当要件

> API通信はすべてHTTPS経由で行われる（US-S06）

### Unit 3 への適用

#### ドメインモデルへの影響

ドメインモデルへの直接的な影響はない。HTTPS はインフラ層（API Gateway / ロードバランサ）の責務。

#### 適用ポイント

- Unit 4 から Unit 3 への PACT 契約に基づく API 呼び出しは HTTPS 経由で行う
- オーナー管理画面から Unit 3 への API 呼び出しも HTTPS 経由で行う
- 設定はインフラ層で対応し、ドメインモデルでは考慮不要

## 4. 個人情報の保護

### 該当要件

> 顧客の個人情報（LINEユーザーID、表示名等）は暗号化して保存される（US-S06）

### Unit 3 への適用

#### ドメインモデルへの影響

Unit 3 は顧客の個人情報を直接保持しない。管理するデータはオーナーのスケジュール情報（営業時間、休業日、スロット）であり、顧客個人を特定する情報を含まない。

#### 留意事項

- Slot エンティティの reservationId は予約の識別子であり、顧客個人を直接特定する情報ではない
- ownerId はオーナーの識別子であるが、これ自体は個人情報の暗号化対象とは別に管理される（Unit 1 の認証基盤の責務）

---

## 5. 認証トークン検証（Unit 1 との依存）

### 適用ポイント

Unit 3 の API を呼び出す際、認証トークンの検証が必要。

#### オーナー管理画面からのアクセス

- Unit 1 が発行した認証トークンを検証し、ownerId を取得する
- 取得した ownerId を使って、対象オーナーのデータのみにアクセスを制限する

#### Unit 4 からのアクセス（PACT 契約）

- サービス間通信の認証方式（サービストークンまたはユーザートークンの伝搬）はアーキテクチャ層で決定する
- ownerId はリクエストパラメータとして明示的に渡される（PACT: query.ownerId）

#### ドメインモデルへの影響

- ドメインモデル自体は認証ロジックを含まない
- ownerId をドメインモデルが受け取る時点で、認証・認可は完了している前提
- ドメインモデルは ownerId によるデータ分離のみを責務とする

---

## まとめ

| 横断的関心事 | Unit 3 での対応レベル | 影響する層 |
|-------------|---------------------|-----------|
| マルチテナント分離 | 全集約で ownerId による分離を実施 | ドメイン / アプリケーション / インフラ |
| 排他制御（二重予約防止） | DailySlotList の楽観的ロック + Slot の状態遷移ルール | ドメイン / インフラ |
| HTTPS 通信 | ドメインモデルへの影響なし | インフラ |
| 個人情報保護 | 顧客個人情報を保持しないため直接的な影響なし | - |
| 認証トークン検証 | ドメインモデル外で対応。ownerId のみドメインモデルに渡る | アプリケーション / インフラ |
